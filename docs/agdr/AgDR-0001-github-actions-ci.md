# AgDR-0001 — GitHub Actions CI pipeline

> In the context of a Flutter/Dart app plus a Firebase Cloud Functions
> TypeScript package living in one repo with no existing CI, facing the need
> for automated quality gates on every PR and push to `main`, I decided to add
> a single GitHub Actions workflow (`.github/workflows/ci.yml`) with two
> independent jobs (`flutter`, `functions`) and pinned tool versions, to get
> fast, deterministic, parallel feedback, accepting that the Flutter version
> pin needs periodic manual bumps and that the functions job uses `npm install`
> (not `npm ci`) because the lockfile is intentionally gitignored.

Refs: issue #6.

## Context

- The repository had **no `.github/workflows/`** at all — zero CI.
- Two distinct toolchains in one repo:
  - Flutter/Dart app at the root (`pubspec.yaml`, `environment: sdk: ^3.9.2`).
  - Firebase Cloud Functions TypeScript package under `functions/`
    (`functions/package.json`, `engines.node: "20"`).
- The unit test suite lives under `test/`; integration tests live under
  `integration_test/` and require a device/emulator. Wiring integration tests
  is explicitly **out of scope** (tracked under issue #5).
- The root `.gitignore` intentionally ignores `functions/package-lock.json`
  with the comment *"Cloud Function lockfile (regenerated by npm)"* — there is
  no committed lockfile to install from.
- Issue #3 will bump `engines.node` from 20 to 22; the CI should track that
  change with a single edit, not a hunt-and-replace.

## Options Considered

### CI platform

| Option | Pros | Cons |
|--------|------|------|
| **GitHub Actions** | Native to the GitHub repo, zero extra accounts, free tier for the project size, first-class Flutter + Node community actions, PR status checks out of the box | Vendor-tied to GitHub; YAML expression quirks |
| CircleCI / GitLab CI / Travis | Mature, some richer caching primitives | Extra account + secrets setup, second source of truth, no benefit at this scale |
| Local pre-commit hooks only | Zero infra | Not enforced on PRs, easily bypassed, no shared visibility |

Chosen: **GitHub Actions** — the repo is on GitHub, the team already reviews
PRs there, and the marketplace actions (`subosito/flutter-action`,
`actions/setup-node`) cover both toolchains with no custom infra.

### Job topology

| Option | Pros | Cons |
|--------|------|------|
| **Two independent jobs (flutter, functions)** | Parallel → faster; a Flutter failure doesn't mask a functions failure (and vice versa); clear separation of concerns | Two checkout steps (cheap) |
| One job doing both | Single setup | Serial (slower); first failure hides the other side; mixed toolchain setup in one job is harder to read |
| Many fine-grained jobs (analyze / test / build / lint each) | Granular status | Setup duplicated N times; over-engineered for current size |

Chosen: **two independent jobs**. No `needs:` between them — they run in
parallel and report independently. This matches the natural seam in the repo
(app vs. functions) and gives the fastest, clearest signal.

### Version pinning

| Option | Pros | Cons |
|--------|------|------|
| **Pin Flutter `3.35.7` + Node from a single env constant** | Deterministic, reproducible builds; satisfies `sdk: ^3.9.2` exactly; single-line bumps | Flutter pin needs a manual bump for new Flutter releases |
| Floating `channel: stable` (no version) | Auto-tracks latest stable | Non-deterministic; a new stable could silently change Dart/analyzer behaviour and break CI unrelated to the PR |
| Hardcode Node `20` inline in the step | Simple | Drifts from `engines.node`; issue #3's bump becomes a multi-spot edit |

Chosen: **pin Flutter to a concrete patch (`3.35.7`)** and **express Node as a
single `NODE_VERSION` env constant** documented as mirroring
`functions/package.json` → `engines.node`.

Flutter `3.35.7` is the latest stable (2025-10-23) that bundles **Dart 3.9.2**,
which exactly satisfies `pubspec.yaml`'s `environment: sdk: ^3.9.2`. This is
load-bearing: Flutter `3.35.0`–`3.35.2` bundle Dart `3.9.0`, which does **not**
satisfy `^3.9.2`. A floating `3.35.x` could therefore resolve to a Dart that
fails `flutter pub get`. The patch pin is intentional, not incidental.

## Decision

Chosen: **a single `.github/workflows/ci.yml` with two parallel jobs and
pinned tool versions**, because it gives deterministic, fast, independent
feedback on both toolchains with the least infrastructure, and the version
strategy keeps the Node bump (issue #3) to a one-line change.

Concrete wiring:

- **`flutter` job:** `actions/checkout@v4` → `subosito/flutter-action@v2`
  (channel `stable`, `flutter-version: 3.35.7`, cache on) → `flutter pub get`
  → `flutter analyze` (fails the job on any analyzer issue — default
  behaviour) → `flutter test test/` (unit suite only).
- **`functions` job:** `actions/checkout@v4` → `actions/setup-node@v4`
  (`node-version` from the `NODE_VERSION` env constant) → `npm install`
  (working dir `functions/`) → `npm run build` → `npm run lint --if-present`.

## Consequences

- Every PR and every push to `main` runs both jobs in parallel; both must be
  green before merge once branch protection requires them.
- The Flutter version pin must be bumped manually when the project adopts a
  newer Flutter/Dart. The inline comment in `ci.yml` flags the Dart-3.9.2
  constraint so a future bumper doesn't accidentally regress below it.
- Node version is bumped in **one place** (`NODE_VERSION` env in `ci.yml`)
  when issue #3 lands; the comment points back to `engines.node` as the
  source of truth.
- **Tradeoff — `npm install` instead of `npm ci`:** because
  `functions/package-lock.json` is gitignored by repo decision, the functions
  job cannot use `npm ci` (which requires a committed lockfile). `npm install`
  resolves the dependency tree fresh each run. This is slightly less
  reproducible and slightly slower than a lockfile-backed `npm ci`, but it is
  the only correct option given the existing repo convention. Revisiting this
  (committing a lockfile) is a separate decision outside issue #6's scope.
- Integration tests remain unwired (issue #5). The `flutter` job is scoped to
  `test/` and the YAML carries an explicit follow-up note so the boundary is
  not silently crossed.
- No repository secrets are required by this pipeline (no deploy, no signing,
  no emulator credentials). Deploy/signing live in `scripts/` and are out of
  scope here.
